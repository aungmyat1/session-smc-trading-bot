# Execution Layer Gap Analysis

`SYSTEM2_MASTER_PLAN.md` and `docs/systems/system2/STATUS.md` are unusually rigorous self-audits and their claims largely hold up against direct code inspection. This report refines rather than contradicts them, and traces every rating to specific files/functions actually read, not the docs' own narrative.

| # | Dimension | Rating | Evidence | Gap before demo validation is credible |
|---|---|---|---|---|
| 1 | Broker connectivity | **PARTIAL** | `execution/mt5_connector.py` (the live path) has `connect()`, guarded `reconnect()`, `ensure_connected()`/`heartbeat()` liveness checks (10s timeout), reconnect counters. | Three parallel broker-client implementations coexist: `mt5_connector.py` (live), `execution/metaapi_client.py` (used by a separate dormant stack — `bot.py`, `execution/order_manager.py`, `adaptive/run_shadow.py`), and an unused `execution/mt5_executor.py`. Consolidate or explicitly retire the dormant ones before trusting "the broker layer" as singular. |
| 2 | Order lifecycle | **PARTIAL** | Clean intent→fill flow: `production/engine/execution_pipeline.py`'s `ExecutionIntent`/`AdapterResult` drives `TradeManager.open_position()` → `execution/execution_state.py`'s state machine (`SIGNAL_RECEIVED → RISK_APPROVED → SUBMISSION_PENDING → BROKER_ACKNOWLEDGED/RECOVERY_PENDING → ...`). `execution/startup_recovery.py` structurally never resubmits on crash-recovery. | `TradeManager.open_position()` (`execution/trade_manager.py:78`) generates a fresh `idempotency_key` per call including a random UUID suffix — meaning the key is never checked against an existing record before creating a new one. Idempotency is real only at *crash-recovery* time; there is no live dedup-by-key guard against an accidental double-call during normal operation (the only practical guard there is `PortfolioManager`'s one-open-position-per-symbol lock, a different mechanism entirely). |
| 3 | Retries | **DONE** (order placement) / **PARTIAL** (everything else) | `execution/trade_manager.py::_place_order_with_retry()` — 3 attempts, exponential backoff, classifies transient/ambiguous/terminal errors, records each attempt. | `get_positions()`/`get_account_info()` calls have no retry loop anywhere — a transient failure raises once, caught by a bare `try/except` at the call site, never retried with backoff. Only order *placement* has real retry logic; reads do not. |
| 4 | Reconciliation | **DONE** (mostly) | Three layers, both directions covered: (a) `execution/close_reconciliation.py`/`position_close_detector.py` run **every tick** (journal-thinks-open-but-broker-closed direction); (b) `scripts/reconcile_positions.py`, scheduled every 5 min via systemd timer, explicitly reports `orphan_broker_positions` (the other direction) and `stale_db_trades`; (c) `startup_recovery.py` runs once at process start. | Coverage is real but split across three mechanisms with different cadences (per-tick / 5-min / startup-only) rather than one unified loop — drift between ticks is possible for anything not covered by the per-tick close-detector specifically. |
| 5 | Emergency stop | **PARTIAL** | Verified directly in `scripts/run_st_a2_demo.py::_tick()`: every tick loads `control_state.json`, and if `emergency_stop.active`, calls `emergency_close_all()`, marks the run `"blocked"`, and **returns before any new-signal/order code runs** — a real, per-tick, order-blocking check, not just a dashboard reflection. This session's fixes (DB-commit correctness, resume-scoping, kill-switch routing — see `docs/github/PR22_FIX_REPORT.md`) are additive to this. | This wiring exists **only** in the deployed runner. `run_portfolio.py` has no such check at all — currently mitigated solely by `RUN_PORTFOLIO_ALLOW_START` blocking it from starting, not by an in-loop control. If that script is ever activated without also adding this check, the emergency stop would not protect it. |
| 6 | Logging | **PARTIAL** | `execution/operations_recorder.py` writes structured rows to Postgres (`operations.execution_event`/`intent`/`risk_decision`/`order_record`/`recovery_checkpoint`, migration 004 applied) plus a JSONL fallback. | The recorder is deliberately best-effort/fire-and-forget — a DB write failure is logged and swallowed, with no signal back to the caller. Several ORM tables (`fill`, `position_record`, `market_data_health`, `reconciliation`) exist with no writer yet. |
| 7 | Monitoring | **PARTIAL** | `dashboard/status_server.py` exposes `/api/operations/*`, `/metrics` (Prometheus), `/api/v1/production/health`. `monitoring/telegram.py` alerts on reconnect failures, watchdog-critical, emergency stop, and reconciliation mismatches — not just trade closes. `smc-demo-runner.service` has `Restart=always`. | `/api/v1/production/health`'s heartbeat file is never written by the canonical runner — a liveness check against that endpoint would show stale/UNKNOWN data, not an honest "I don't know." |
| 8 | Recovery behavior | **DONE** (deployed runner) / **MISSING** (`run_portfolio.py`) | `execution/startup_recovery.py::reconcile_pending_executions()` resolves crash-mid-submission purely from broker truth, never resubmits, flags orphaned positions; called from `run_st_a2_demo.py::run()` before the tick loop starts. Tests exist (`tests/execution/test_startup_recovery.py`, `tests/scripts/test_run_st_a2_demo_e2e_recovery.py`). `Restart=always`/`RestartSec=15` confirms this path is actually exercised on crash/reboot, not just in tests. | `run_portfolio.py` never calls the recovery function at all (confirmed via grep — no call site). Recovery test fixtures use synthetic broker-position dicts, not a real MetaAPI session shape — the reconciliation logic itself has never been exercised against a live broker response. |

## Overall assessment

The **deployed path** (`scripts/run_st_a2_demo.py` / `smc-demo-runner.service`) is materially more complete than the historical framing in older docs suggested — broker connectivity, retry-with-backoff on order placement, per-tick reconciliation, per-tick emergency-stop enforcement, and crash-recovery are all real and wired into the process that actually trades today. The **undeployed** `run_portfolio.py` (`CanonicalExecutionPipeline`/`RiskFirewall` architecture) lacks nearly all of this safety wiring and is currently hard-blocked from starting by default — an appropriate mitigation, but it leaves a second, materially less-safe implementation sitting in the repo.

**The two findings that most directly affect demo-validation credibility**, not previously documented:
1. Idempotency protection is real only at crash-recovery time (never-resubmit-on-restart), not as a live dedup-by-key guard against an accidental double order during normal operation.
2. Three separate broker-client implementations coexist even though only one is on the live path — anyone reasoning about "the broker layer" from code alone would need to know which one is live without being told.
