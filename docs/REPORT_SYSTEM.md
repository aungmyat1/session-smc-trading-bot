# Report System

This document defines the operating report system for the project. Reports are
owner-facing artifacts saved under `reports/` and generated by:

```bash
python scripts/generate_reports.py --type <report-type>
```

## Storage Layout

```text
reports/
├── daily/
├── weekly/
├── monthly/
├── strategy/
├── risk/
├── execution/
├── system_health/
├── incidents/
└── live_readiness/
```

## Daily Trading Report

Purpose:
Summarize the latest operating and trading session.

Input data:
`logs/*.jsonl`, `logs/bot_state.json`, `logs/bot.log`, `logs/st_a2_runner.log`, config files.

Metrics:
Date, mode, opened/closed trades, win rate, Net R, PF, drawdown, rejected
signals, risk status, database status, broker status, critical incidents, final
recommendation.

Output format:
Markdown.

When to generate:
End of trading day, or after a major interruption.

Where saved:
`reports/daily/`

How the owner should use it:
Decide whether the next session should continue, pause, or be reviewed.

## Strategy Performance Report

Purpose:
Evaluate the active strategy over a wider evidence window.

Input data:
Strategy catalog, journals, trade database, execution summaries.

Metrics:
Strategy name, pair, session, trades, win rate, PF, expectancy, drawdown, best
and worst setup, robustness status, recommendation.

Output format:
Markdown.

When to generate:
Weekly, monthly, and before any promotion decision.

Where saved:
`reports/strategy/`

How the owner should use it:
Decide whether to promote, continue demo, retire, or gather more data.

## Risk Report

Purpose:
Tell the owner whether the system is within its operating loss and exposure
limits.

Input data:
`logs/bot_state.json`, recent journals, configured demo risk assumptions.

Metrics:
Exposure, daily risk used, per-trade risk, drawdown, consecutive losses,
emergency stop, breaches, recommendation.

Output format:
Markdown.

When to generate:
Daily and after any loss cluster or halt.

Where saved:
`reports/risk/`

How the owner should use it:
Decide whether the system can continue operating or must remain paused.

## Execution Quality Report

Purpose:
Measure how well the signal-to-order-to-close chain is behaving.

Input data:
`logs/trades.jsonl`, optional `logs/execution_summary_*.json`.

Metrics:
Latency, fills, rejects, fill rate, reject reasons, slippage coverage,
reconnect evidence.

Output format:
Markdown.

When to generate:
Daily after activity, weekly during demo review, and after execution incidents.

Where saved:
`reports/execution/`

How the owner should use it:
Distinguish strategy issues from execution-layer issues.

## Database Health Report

Purpose:
Confirm the data and journal layer is usable.

Input data:
Runtime DB configuration, journal/database files, recovery checks.

Metrics:
Connection status, backend detail, recovery state, file presence.

Output format:
Markdown.

When to generate:
After startup problems, data issues, or before a restart decision.

Where saved:
`reports/system_health/`

How the owner should use it:
Decide whether the platform has the data integrity needed to operate safely.

## System Health Report

Purpose:
Provide a single operational health verdict.

Input data:
Runner status, DB health, risk and recovery checks, recent logs.

Metrics:
Database connection, broker/API connection, worker status, log errors, missing
data, heartbeat/recovery status, last successful run, critical alerts.

Output format:
Markdown.

When to generate:
Daily, before restart, and during incident handling.

Where saved:
`reports/system_health/`

How the owner should use it:
Decide whether the platform is healthy enough to continue operating.

## Incident Report

Purpose:
Record and review operational failures or instability.

Input data:
Recent log tails, health checks, recovery status.

Metrics:
Incident count, critical incident count, affected status signals, recent event
lines.

Output format:
Markdown.

When to generate:
Immediately after a failure, pause, or suspicious runtime event.

Where saved:
`reports/incidents/`

How the owner should use it:
Keep a durable record of what happened and what must be fixed before resuming.

## Weekly Review Report

Purpose:
Summarize the week from a business and operating perspective.

Input data:
Daily report sources, journals, incidents, strategy trend.

Metrics:
Closed trades, win rate, Net R, PF, drawdown, signals, rejections, incident
count, action prompts.

Output format:
Markdown.

When to generate:
End of week.

Where saved:
`reports/weekly/`

How the owner should use it:
Review whether the current strategy and operating mode should continue.

## Monthly Strategy Review Report

Purpose:
Support higher-level lifecycle decisions.

Input data:
Strategy metrics, validation state, live-readiness evidence.

Metrics:
Monthly results, best/worst setups, robustness state, readiness verdict.

Output format:
Markdown.

When to generate:
End of month or before any promotion review.

Where saved:
`reports/monthly/`

How the owner should use it:
Decide whether to promote, freeze, revalidate, or retire the strategy.

## Live Readiness Report

Purpose:
State whether the system is ready for demo or live consideration.

Input data:
Strategy catalog, validation evidence, risk state, system health, incidents.

Metrics:
Validation gate, paper trading result, execution stability, database stability,
risk firewall, governance, emergency stop test, incident count, final verdict.

Output format:
Markdown.

When to generate:
Before any demo or live promotion decision, and after major incidents.

Where saved:
`reports/live_readiness/`

How the owner should use it:
Make a go/no-go decision without relying on memory or scattered notes.

## CLI Usage

```bash
python scripts/generate_reports.py --type daily
python scripts/generate_reports.py --type weekly
python scripts/generate_reports.py --type monthly
python scripts/generate_reports.py --type strategy
python scripts/generate_reports.py --type risk
python scripts/generate_reports.py --type execution
python scripts/generate_reports.py --type system-health
python scripts/generate_reports.py --type incident
python scripts/generate_reports.py --type live-readiness
python scripts/generate_reports.py --type all
```

All generated reports are Markdown files with timestamped filenames.
